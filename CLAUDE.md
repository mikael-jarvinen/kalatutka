# CLAUDE.md

You are working on **Siikasää**, a fishing-conditions forecasting tool for whitefish (siika, *Coregonus lavaretus*) bottom fishing at Saaronniemi, Ruissalo, Turku, Finland (60.4164°N, 22.0939°E).

Read this file first in any new session. It is the project's memory.

---

## What the user wants

The user is a hobbyist shore angler who fishes whitefish with bottom rigs at Saaronniemi. They got frustrated by:

1. Standard fishing calendars (solunar / moon-phase) that don't apply to the Baltic Sea (tidal range under 20 cm).
2. Generic weather-based forecasts that ignore Saaronniemi's specific geometry and the biology of sight-feeding whitefish in cold water.

They want a tool that gives them an honest, biologically grounded probability score for each day, and learns over time as they log catches.

When in doubt, **prefer the user's own shore measurements over any modeled value.** Their reality check is what calibrates this tool.

---

## Domain context — read before touching the scoring

Whitefish in spring at Saaronniemi feed on **polychaete worms (harvasukamadot)** that rise from sandy/gravel bottoms when sun warms the substrate. They are **sight-feeders** with peaks at **10–12** and **17–19** local time. The fishing window is roughly **two weeks after ice-out through mid-May** (water in 4–10 °C range; collapses above 12 °C when roach and perch take over the inshore).

**Saaronniemi geometry:** the cape (Kolkannokka) opens to the **W/SW** with the open Saaristomeri to the south and west. Therefore:

- Onshore wind = wind FROM S, SW, W (180–270°). Best fishing wind is **SW (~225°)**.
- Offshore wind = wind FROM N, NE (0–90°). Worst is NE (45°): cold *and* offshore *and* the shore water leaves.

**The biggest single factor is water temperature**, not air temperature. Air is a poor proxy for shore water (which has thermal lag and heavy local variation in shallow pockets). Always weight water heavily, accept user shore measurements when given, and apply hard penalty multipliers below 4 °C and above 12 °C.

**Pressure level alone does not affect feeding** (the swim-bladder argument is biophysically marginal compared to the pressure changes a fish creates by swimming a meter up or down). The pressure *change rate* over 24–48h is the actual signal — it correlates with weather pattern transitions that have real biological effects (cloud cover, wind, prey activity). Score absolute |Δp|; reward 3–8 hPa/24h, penalize stable (|Δp|<3) and extreme (|Δp|>15).

## Tech stack

- **Frontend:** vanilla HTML + CSS + ES module JavaScript. No framework. No build step.
- **Run:** open `src/index.html` directly, or `python3 -m http.server 8000 --directory src`.
- **Data:** snapshot in `src/data/data.js`, regenerated by `scripts/refresh.mjs`. Three sources: Open-Meteo Weather, Open-Meteo Marine, FMI Open Data WFS (Föglö Degerby mareograph). In production, the docker-compose `refresh` service runs the script on a loop and writes to a shared volume the `web` service serves; locally, use `npm run refresh`.
- **Tests:** none yet. `src/scoring.js` is pure functions specifically so it's testable. Add tests when you change scoring.

The project was originally a single inline HTML artifact in a Cowork session; it has been split into modules to make it work with Claude Code.

## Conventions

- **UI strings: Finnish.** The user is Finnish and this is for personal use. Keep all visible labels in Finnish.
- **Code, comments, docs: English.** Standard for code repos.
- **Domain terms in code:** prefer Finnish where the term is more precise (`siika` not `whitefish`, `harvasukamato` not `polychaete worm`). Add English in comments.
- **Pure functions in `scoring.js`** — no DOM access, no state mutation. They take values, return numbers/objects. This is non-negotiable; tests depend on it.
- **`data/data.js`** exports plain const arrays/objects and is auto-generated. Never compute scores there; that belongs in `scoring.js`. Never edit it by hand; edit `scripts/refresh.mjs`.
- **`app.js`** holds the only mutable state (`state` object) and is the only file that touches the DOM.
- **No frameworks** unless the user explicitly asks. Vanilla works fine for this app's complexity.

## What's intentionally NOT in the model

- **Solunar / moon phase** — doesn't apply to Baltic. If you add it, you're wrong.
- **Absolute barometric pressure** — only the change matters.
- **Generic "fishing calendar" data** — those tools don't know about whitefish or the Baltic.
- **Live API calls from the page** — by design the tool runs from a baked-in snapshot. A refresh script (or backend) is the right place to fetch.

## Common tasks

### Update the data snapshot

`src/data/data.js` is **auto-generated** by `scripts/refresh.mjs`. Don't edit
it by hand — your changes will be overwritten the next time the refresh runs.

- **Local one-shot:** `npm run refresh` (writes into `src/data/`).
- **Dry run:** `npm run refresh:dry` (writes to `/tmp/siikasaa-dryrun/`).
- **In production:** the docker-compose `refresh` service runs the script on
  a `REFRESH_INTERVAL_HOURS` loop (default 4h). See `deploy/README.md`.

To change *what* gets fetched (e.g. add a new field), edit `scripts/refresh.mjs`
and the `data.js` template inside it. To change scoring math, edit
`src/scoring.js` and `docs/methodology.md`.

### Tweak the scoring weights or thresholds

Edit `src/scoring.js` only. Document the change in `docs/decisions.md` with the rationale and any user observation that motivated it. If the change is non-trivial, validate against the user's catch log (when one exists) or against known good/bad days they've reported.

### Add a new factor to the score

1. Add a `score<Factor>(value)` pure function to `src/scoring.js`.
2. Add the factor to the `factors` array in `scoreDay()` with a weight. Other weights may need to shrink so total weight stays at 100.
3. Add a critical penalty multiplier in `scoreDay()` if applicable.
4. Update `docs/methodology.md` (single source of truth).
5. If the factor needs new input data, update `scripts/refresh.mjs` (so the new field gets fetched and written into `src/data/data.js`) and `docs/data-sources.md`.

### Add unit tests

There's no test runner set up yet. Recommended: `vitest` (works with vanilla ES modules). Test the scoring functions only — don't test the rendering layer.

```bash
npm init -y
npm install -D vitest
# create tests/scoring.test.js
npx vitest
```

## Known issues and caveats

- **Single-location.** Hardcoded for Saaronniemi. Multi-location would require per-location wind-direction calibration (each cape has its own geometry).
- **Föglö Degerby is ~50 km SW** of Saaronniemi. It's the closest FMI mareograph with TW data; there is no Turku mareograph. Föglö is directionally useful (trends) but absolute values can differ from Saaronniemi shore by 1–3 °C.
- **Open-Meteo Marine SST is open-water modeled**, runs ~2–3 °C warmer than shore measurements after cold wind events. User's own measurement is always more reliable.
- **No history.** Each session starts fresh; no localStorage usage. If you add persistence, use a backend or IndexedDB; do not use localStorage in the artifact context (was the original constraint, no longer applies in a normal web app — but document the choice).
- **Hourly water temp not modeled.** We project future-day water temp from air-temp delta with a ~0.2 damping coefficient. This is crude. A better approach uses Open-Meteo Marine forecast directly per future day.
